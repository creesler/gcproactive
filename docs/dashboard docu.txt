Prepaid Credit Converter System
MVP Technical Documentation
1. Purpose
This system converts third-party provider usage into our own internal prepaid credit system.
It is designed for clients who use our automations through their own n8n instance.
Whenever an automation uses a billable provider such as OpenAI, Gemini, Retell, Airtop, or similar, the workflow sends a webhook to our Node.js API.
Our API then:
1.	identifies the client 
2.	reads the provider usage 
3.	converts provider usage into internal credits 
4.	deducts the client balance 
5.	stores the usage event 
6.	returns the remaining balance 
This system is separate from monthly hosting/retainer billing.
________________________________________
2. Billing Model
2.1 Two separate ledgers
A. Prepaid usage credits
Used for variable-cost services such as:
•	LLM usage 
•	AI chat replies 
•	voice call usage 
•	scraping usage 
•	other external paid providers 
B. Monthly service/retainer ledger
Used for:
•	hosting subscription 
•	support hours remaining 
•	warranty period 
•	monthly maintenance 
These two ledgers must remain separate.
________________________________________
3. Core Architecture
3.1 Basic flow
Client n8n workflow:
1.	automation starts 
2.	billable provider node runs 
3.	n8n extracts usage or cost from provider response 
4.	n8n sends usage payload to our Node server 
5.	Node server computes credits used 
6.	Node server deducts credits from client balance 
7.	Node server logs event 
8.	Node server returns result to n8n

6. Tracking Strategy
6.1 Recommended method
Track usage in real time inside n8n.
Do not rely on hourly fetching as the primary billing method.
Why
Real-time event billing is better because:
•	each event is tied to a client 
•	each event is tied to a workflow execution 
•	easier to prevent duplicate charges 
•	easier to explain usage later 
•	easier to stop workflows on insufficient balance 
6.2 Hourly fetching
Hourly fetching may still be used only for:
•	reconciliation 
•	sanity checks 
•	comparing totals against provider dashboards 
But hourly fetch should not be the main source of truth.

7. n8n Workflow Billing Pattern
7.1 General node pattern
Typical billable workflow structure:
1.	Trigger node 
2.	Client Context node 
3.	Provider Call node 
4.	Usage Normalize node 
5.	Billing Webhook node 
6.	Optional Balance Check / Fallback node 
7.	Continue workflow 
7.2 Node responsibilities
Trigger node
Starts workflow:
•	webhook 
•	schedule 
•	Facebook event 
•	WhatsApp event 
•	WordPress event 
•	form submission 
Client Context node
A Set node near the start that provides:
•	client_id 
•	workflow_name 
•	channel 
•	optional project/agent/source identifiers 
Provider Call node
The actual paid service call:
•	OpenAI 
•	Gemini 
•	Chatbase 
•	Retell 
•	Airtop 
Usage Normalize node
Transforms provider-specific response format into one common internal payload.
Billing Webhook node
HTTP POST to our Node.js API.
Optional fallback node
Used if:
•	insufficient credits 
•	low balance 
•	billing API error 
________________________________________
8. What should trigger billing
Billing webhook should be called only after nodes that create billable provider usage.
Billable nodes
•	LLM response generation 
•	chatbot reply generation 
•	voice call completion 
•	scrape task completion 
•	browser session completion 
•	transcription 
•	OCR 
•	other external paid services 
Non-billable nodes
Do not bill for:
•	IF nodes 
•	Switch nodes 
•	Merge nodes 
•	Set/Edit nodes 
•	formatting nodes 
•	logic-only Code nodes 
•	internal database reads 
•	plain routing logic

10. Pricing and Conversion Rules
10.1 Main rule
Whenever possible, compute billing from raw provider cost.
Universal formula:
credits_used = ceil((raw_cost * markup_multiplier) / credit_value)
10.2 Example
If:
•	raw_cost = 0.018 USD 
•	markup_multiplier = 4 
•	credit_value = 0.01 USD 
Then:
•	billable value = 0.072 USD 
•	credits_used = ceil(7.2) 
•	final charge = 8 credits 
10.3 If raw cost is not sent
Then Node server computes raw cost from provider usage units.
Examples:
•	input_tokens and output_tokens 
•	minutes 
•	scrape credits 
•	sessions 
•	API calls 
________________________________________
11. Pricing Rule Types
11.1 Cost-based rule
Used when provider raw cost is available.
Example:
•	provider: airtop 
•	service: scraping 
•	rule: use raw_cost × markup 
11.2 Token-based rule
Used when provider returns token counts but not raw cost.
Example:
•	provider: openai 
•	model: gpt-4o-mini 
•	input token price per 1M 
•	output token price per 1M 
Node server computes raw cost first.
11.3 Unit-based rule
Used for things like:
•	voice minutes 
•	call duration 
•	scrape runs 
•	browser sessions

13. Duplicate Protection
13.1 Idempotency
Every usage event must include a unique event_id.
If the same event_id is received more than once:
•	do not charge again 
•	return existing result or ignore duplicate 
This protects against:
•	n8n retries 
•	webhook resends 
•	network errors 
________________________________________
14. Security
Each client n8n instance should authenticate its calls to the Node billing API.
Recommended:
•	per-client API key 
•	HMAC signature or secret header 
•	HTTPS only 
Minimum headers:
•	x-client-id 
•	x-api-key 
Optional:
•	x-signature

20. Key Design Recommendation
Use this principle:
n8n tracks usage per event, Node server computes billing, database stores truth.
Do not use hourly provider fetch as the main billing mechanism.
Use hourly fetch only for reconciliation.
________________________________________
21. Final Summary
This MVP is a universal prepaid conversion engine.
It allows:
•	multiple providers 
•	one internal credit language 
•	client-isolated billing 
•	separate monthly retainer handling 
•	easy future scaling 
The simplest successful version is:
•	n8n sends usage events 
•	Node computes cost and credits 
•	client balance updates instantly 
•	monthly hosting/support remains separate
